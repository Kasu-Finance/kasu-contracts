Executive Summary

Overall Risk Assessment:

MEDIUM

The Kasu protocol demonstrates strong security engineering with comprehensive role-based access control, proper use of OpenZeppelin v5 libraries, epoch-based design that eliminates most MEV vectors, and defense-in-depth patterns throughout. This second audit, conducted using three complementary methodologies (Feynman deep logic analysis, State Inconsistency structural analysis, and Nemesis iterative fusion), confirms the protocol's solid architectural foundation while surfacing one high-severity state inconsistency bug and several medium-severity issues warranting attention.

Top Risks

  1. UserManager stale _isUserPartOfLendingPool mapping corrupts re-enrollment — After pool removal, the mapping is never reset to false, causing re-depositing users to be invisible to loyalty calculations and potentially gaining unfair clearing priority (H-01).
  2. Fee emission permanently lost when no eligible rKSU holders — USDC fees transferred to KSULocking when eligibleRKSUForFees == 0 are permanently stranded with no recovery mechanism (M-01).
  3. Admin privilege risks without guardrails — recoverERC20 can drain KSU reward pools, emitUserLoyaltyRewardBatch allows arbitrary reward inflation, config changes take immediate effect without timelock (M-02, M-03, M-08).
  4. Loss repayment has no upper bound — recoveredAmount can exceed lossAmount, enabling over-recovery that distributes excess funds to loss token holders (M-05).
  5. Balance-based swap accounting — DepositSwap._transferAndSwap uses absolute balanceOf instead of delta, potentially crediting residual tokens to swap depositors (M-07).

Methodology

This audit employed three complementary methodologies executed in parallel, with a final iterative fusion pass:

1. Feynman Deep Logic Audit

First-principles reasoning applied to every function. Questions every line's purpose, ordering, guards, and implicit assumptions. Finds business logic bugs that pattern-matching and checklists miss. Applied across 3 parallel workstreams covering all 42 contracts.

2. State Inconsistency Audit

Structural invariant analysis mapping all coupled state pairs, building mutation matrices, and cross-checking every mutation path for missing state synchronization. 8-phase process covering dependency mapping, parallel path comparison, user journey tracing, and masking code detection. 15 coupled state pairs mapped, ~50 mutation paths analyzed.

3. Nemesis Fusion Pass

Iterative cross-pollination of Feynman and State Inconsistency findings. Tests intersection hypotheses where bugs from different methodologies compound. 3 intersection hypotheses investigated; 2 confirmed as true intersections (non-amplifying), 1 eliminated.

High Severity Findings

H-01 HIGH Stale _isUserPartOfLendingPool mapping corrupts user re-enrollment and clearing priority
CONTRACT UserManager.sol — _removeLendingPoolFromUser() (lines 497–499)
METHODOLOGY State Inconsistency Audit (SI-001) + Feynman Audit (F-02) — cross-confirmed

Impact

When a lending pool is removed from a user's _userLendingPools array via swap-and-pop in _removeLendingPoolFromUser(), the _isUserPartOfLendingPool[lendingPool][user] mapping is never reset to false. There is no code path in the entire codebase that sets this mapping to false. If the user later deposits into the same pool, the guard at line 314 incorrectly skips re-adding the pool to _userLendingPools[user], leaving the array empty.

Exploit Scenario

  1. Alice deposits into Pool X. _isUserPartOfLendingPool[X][Alice] = true, _userLendingPools[Alice] = [X].
  2. Alice fully withdraws. Admin calls updateUserLendingPools(). Pool X is removed from Alice's array via swap-and-pop, but _isUserPartOfLendingPool[X][Alice] remains true.
  3. Alice deposits into Pool X again. userRequestedDeposit() checks _isUserPartOfLendingPool[X][Alice] — it's true, so the push is skipped. _userLendingPools[Alice] remains empty.
  4. _userTotalPendingAndActiveDepositedAmount(Alice) iterates the empty array and returns 0.
  5. If Alice has rKSU: rKSUDepositRatio = type(uint256).max (division by zero guard), giving Alice maximum loyalty level and unfair clearing priority, potentially displacing other users' deposits/withdrawals.
  6. If Alice has no rKSU: her deposit amount is reported as 0 to the loyalty reward system. She receives zero loyalty rewards despite having an active deposit — permanent loss of earned rewards.

Root Cause

Missing _isUserPartOfLendingPool[pool][user] = false in the removal function. The only write to this mapping is = true at line 316.

Recommended Fix

function _removeLendingPoolFromUser(address user, uint256 userLendingPoolIndex) internal {
    address lendingPool = _userLendingPools[user][userLendingPoolIndex];
    _isUserPartOfLendingPool[lendingPool][user] = false;  // ADD THIS LINE
    _userLendingPools[user][userLendingPoolIndex] =
        _userLendingPools[user][_userLendingPools[user].length - 1];
    _userLendingPools[user].pop();
}

Test to Add

User deposits, fully withdraws, updateUserLendingPools is called, user re-deposits into the same pool. Verify pool is tracked in _userLendingPools, loyalty level is correct, and clearing priority is fair.

Medium Severity Findings

M-01 MEDIUM Fee emission permanently lost when no eligible rKSU holders
CONTRACT KSULocking.sol — emitFees() / _updatePoolRewards() (line 427)

Impact

When eligibleRKSUForFees == 0, USDC fees are transferred to KSULocking via safeTransferFrom but _updatePoolRewards() silently returns without updating _accumulatedRewardsPerShare. These tokens are permanently stuck with no recovery mechanism. When users later enable fees, they will never receive these tokens because the per-share accumulator was never incremented.

Root Cause

_updatePoolRewards silently returns when no eligible recipients exist, but tokens were already transferred.

Recommended Fix

Revert in emitFees when eligibleRKSUForFees == 0, or buffer unallocated fees for later distribution, or redirect to protocol fee receiver.

Test to Add

Test emitFees when no users are fee-eligible. Verify funds are not permanently lost.

M-02 MEDIUM recoverERC20 can drain KSU reward pool beyond unclaimed balance
CONTRACT UserLoyaltyRewards.sol — recoverERC20() (lines 196–198)

Impact

The admin can transfer ANY ERC20 from the contract, including KSU tokens backing unclaimed user rewards. No guard prevents withdrawing below totalUnclaimedRewards. Users would be unable to claim their accrued KSU rewards as the ksuBalance check at line 170–173 caps claims at the actual balance.

Root Cause

Missing check: tokenAmount <= balance - totalUnclaimedRewards for the KSU token.

Recommended Fix

Add: if (tokenAddress == address(_ksuToken)) require(tokenAmount <= _ksuToken.balanceOf(address(this)) - totalUnclaimedRewards).

Test to Add

Verify recoverERC20 reverts when attempting to withdraw KSU below the unclaimed rewards threshold.

M-03 MEDIUM emitUserLoyaltyRewardBatch allows arbitrary reward inflation
CONTRACT UserLoyaltyRewards.sol — emitUserLoyaltyRewardBatch() (lines 140–157)

Impact

Admin can call with arbitrary parameters (any user, any amountDeposited, any ksuTokenPrice) without on-chain validation against actual state. A compromised admin multisig could inflate rewards and drain all KSU from the contract.

Root Cause

Batch emission function trusts admin input without on-chain state validation.

Recommended Fix

Add a maximum reward cap per call. Ensure admin is a well-configured multisig with timelock. Document this as an intentional admin privilege.

Test to Add

Test that extreme parameter values produce bounded reward amounts.

M-04 MEDIUM Stale KSU epoch price inflates loyalty levels when updateKsuEpochTokenPrice not called
CONTRACT SystemVariables.sol (lines 235–246) / UserManager.sol (line 444)

Impact

updateKsuEpochTokenPrice() must be called each epoch by an external caller. If not called, ksuEpochTokenPrice remains at its last-updated value. The loyalty level calculation in UserManager._userLoyaltyLevel() uses this stale price. If KSU price has dropped since the last update, users' rKSU appears worth more than it is, inflating loyalty levels and giving unfair clearing priority and higher KSU rewards.

Root Cause

No enforcement that updateKsuEpochTokenPrice() is called every epoch. No automated mechanism or clearing-time integration.

Recommended Fix

Call updateKsuEpochTokenPrice() at the start of batchCalculateUserLoyaltyLevels, or use the live price from _ksuPrice.ksuTokenPrice() directly instead of the epoch-cached value.

Test to Add

Skip updateKsuEpochTokenPrice for several epochs, verify loyalty levels use stale price. Fix and verify live price is used.

M-05 MEDIUM Loss repayment has no upper bound; recoveredAmount can exceed lossAmount
CONTRACT LendingPoolTrancheLoss.sol — repayLoss() (line 140)

Impact

The repayLoss function increments recoveredAmount without checking that it doesn't exceed the original lossAmount. If recoveredAmount > lossAmount, the userClaimableLoss() calculation distributes more than the actual loss to token holders, draining excess USDC from the tranche contract.

Root Cause

No upper bound validation: _lossDetails[lossId].recoveredAmount += amount with no cap.

Recommended Fix

if (_lossDetails[lossId].recoveredAmount + amount > _lossDetails[lossId].lossAmount) {
    revert RecoveryExceedsLoss(lossId);
}

Test to Add

Call repayLoss with cumulative amounts exceeding original loss. Verify it reverts.

M-06 MEDIUM Incomplete loss token minting blocks clearing step 4
CONTRACT LendingPoolTrancheLoss.sol (modifier notPendingLossMint) + LendingPoolTranche.sol (lines 139, 170)

Impact

When a loss is reported with doMintLossTokens = false, the notPendingLossMint modifier blocks all deposit() and redeem() on the tranche until batch minting completes. Clearing step 4 calls these functions via acceptDeposit / acceptWithdrawal, causing clearing to revert with LossMintingInProgress. For tranches with many users, minting may take multiple transactions.

Root Cause

ClearingCoordinator.doClearing does not check for pending loss mints before attempting step 4. The reportLoss function requires verifyClearingNotPending, but there is a window between loss report and next clearing where minting must complete.

Recommended Fix

Consider having the clearing coordinator check for pending loss mints and either complete them or skip the affected tranche. Alternatively, require doMintLossTokens = true for production usage and document operational guidelines.

Test to Add

Report loss with doMintLossTokens = false, attempt clearing. Verify clearing reverts. Complete minting and verify clearing proceeds.

M-07 MEDIUM Balance-based swap accounting includes residual tokens
CONTRACT DepositSwap.sol — _transferAndSwap() (line 96)

Impact

The function measures swap output as IERC20(outToken).balanceOf(address(this)) after the swap. If LendingPoolManager holds residual USDC from a prior transfer or donation, that amount is included in the deposit, allowing a user to deposit more than their swap actually produced.

Root Cause

Absolute balance check instead of delta-based accounting (balance before vs after).

Recommended Fix

Record balanceBefore before the swap. Compute outTokenAmount = balanceOf(this) - balanceBefore.

Test to Add

Send USDC directly to LendingPoolManager, then execute a swap deposit. Verify only the swapped amount is deposited.

M-08 MEDIUM Withdrawal pro-rata truncation to zero permanently delays small positions
CONTRACT AcceptedRequestsExecution.sol — line 178

Impact

When computing accepted withdrawal shares pro-rata (userWithdrawalShares * totalAcceptedWithdrawalShares / totalRequestedWithdrawalShares), small positions can truncate to 0. The withdrawal NFT survives with full shares amount, and the user gets 0 shares accepted. This repeats every epoch since the ratio doesn't change for that user's position — the withdrawal request is effectively stuck permanently.

Root Cause

Integer division truncation with no minimum acceptance threshold or dust-clearing mechanism.

Recommended Fix

Either set a minimum accepted amount (e.g., accept fully if shares < dust threshold), or round up for the smallest positions, or allow users to cancel and re-submit.

Test to Add

Create a withdrawal request with a very small share amount relative to total requested. Verify it eventually gets processed (not stuck at 0 forever).

Notable Methodology Difference

February audit H-01/H-02 (FTD batch array corruption): The previous audit flagged swap-and-pop during reverse iteration as HIGH severity. This audit's State Inconsistency analysis (Phase 5: Parallel Path Comparison) specifically investigated this pattern and determined it is correct: reverse iteration processes the last element first, so when an earlier element is removed and the last element swaps in, it has already been processed. The nextIndexToProcess is re-initialized at each clearing start via _initializeLendingPoolFixedTermDepositsProcessing. We classify the original H-01/H-02 as false positives. However, we recommend the Kasu team validate this conclusion with a targeted test case.
