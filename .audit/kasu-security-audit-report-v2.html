Executive Summary

Overall Risk Assessment: MEDIUM

The Kasu protocol demonstrates strong security engineering with comprehensive role-based access control, proper use of OpenZeppelin v5 libraries, epoch-based design that eliminates most MEV vectors, and defense-in-depth patterns throughout. This audit was conducted using three complementary methodologies (Feynman deep logic analysis, State Inconsistency structural analysis, and Nemesis iterative fusion), identifying 1 high, 8 medium, 6 low, and 7 informational findings.

Five findings have been resolved during this audit cycle (H-01, M-01, M-02, M-05, M-07). All fixes have been verified as correct and complete with dedicated test coverage. Four medium-severity findings remain unresolved and are tracked for the next audit cycle.

Top Risks

  1. Admin privilege risks without guardrails — emitUserLoyaltyRewardBatch allows arbitrary reward inflation. All config changes take immediate effect without timelock (M-03).
  2. Stale KSU epoch price inflates loyalty levels — If updateKsuEpochTokenPrice() is not called each epoch, users get inflated loyalty levels and unfair clearing priority (M-04).
  3. Incomplete loss token minting blocks clearing — Reporting a loss with doMintLossTokens = false can block clearing step 4 until all loss tokens are minted (M-06).
  4. Withdrawal pro-rata truncation permanently delays small positions — Integer division truncation to 0 causes small withdrawal requests to be stuck indefinitely (M-08).

Resolved Findings

The following findings were identified during this audit and resolved before report finalization. All fixes have been verified as correct and complete with dedicated test coverage (195 tests passing).

H-01 RESOLVED Stale _isUserPartOfLendingPool mapping corrupts user re-enrollment and clearing priority
Contract

UserManager.sol — _removeLendingPoolFromUser()

Impact

When a lending pool was removed from a user’s _userLendingPools array via swap-and-pop, the _isUserPartOfLendingPool[pool][user] mapping was never reset to false. Re-depositing into the same pool skipped re-adding it, leaving the user’s pool list empty. Users with rKSU got type(uint256).max loyalty ratio (unfair clearing priority); users without rKSU got zero loyalty rewards.

Fix Applied

Added _isUserPartOfLendingPool[_userLendingPools[user][idx]][user] = false; before the swap-and-pop. The mapping is read before the swap overwrites the element. Pattern matches existing _removeUserFromAllUsers which follows the same set-false-then-swap approach.

Verification

2 dedicated tests added: test_H01_reDepositAfterCleanup_poolMustBeReTracked and test_H01_loyaltyLevelCorrectAfterReDeposit. Both pass. Full suite: 195/195.

M-01 RESOLVED Fee emission permanently lost when no eligible rKSU holders
Contract

KSULocking.sol — emitFees() / FeeManager.sol — emitFees()

Impact

When eligibleRKSUForFees == 0, USDC fees were transferred to KSULocking but _accumulatedRewardsPerShare was not updated, permanently stranding the fees.

Fix Applied

Two-layer defense: (1) KSULocking.emitFees() reverts with NoEligibleFeeRecipients() when no eligible holders. (2) FeeManager.emitFees() checks eligibleRKSUForFees > 0 before calling KSULocking; when false, redirects all fees to protocol fees. No dangling approval left when skipping.

Verification

2 dedicated tests added. Base mainnet deployment confirmed not affected (FeeManager already had the guard; no stranded USDC). Full suite: 195/195.

M-02 RESOLVED recoverERC20 can drain KSU reward pool beyond unclaimed balance
Contract

UserLoyaltyRewards.sol

Fix Applied

Removed recoverERC20 function entirely from UserLoyaltyRewards.sol, UserLoyaltyRewardsLite.sol, and IUserLoyaltyRewards.sol. No references remain in src/.

Verification

1 dedicated test confirms function no longer exists via low-level call. Full suite: 195/195.

M-05 RESOLVED Loss repayment has no upper bound; recoveredAmount can exceed lossAmount
Contract

LendingPoolTrancheLoss.sol — repayLoss()

Fix Applied

Added check: if (recoveredAmount + amount > lossAmount) revert RecoveryExceedsLoss(lossId) before incrementing. Uses > (not >=) so exact-amount repayment succeeds. No overflow concern — Solidity 0.8.23 has built-in overflow checks.

Verification

1 dedicated test confirms exact repayment succeeds and 1 additional wei reverts. Full suite: 195/195.

M-07 RESOLVED Balance-based swap accounting includes residual tokens
Contract

DepositSwap.sol — _transferAndSwap()

Fix Applied

Changed from absolute balanceOf(this) to delta-based accounting: capture balanceBefore before swap, compute outTokenAmount = balanceOf(this) - balanceBefore after. If the swap somehow reduces balance, Solidity 0.8.23 reverts on underflow (correct behavior).

Verification

Fix verified via code trace. Full suite: 195/195.

Medium Severity Findings (Remaining)

M-03 MEDIUM emitUserLoyaltyRewardBatch allows arbitrary reward inflation
Contract

UserLoyaltyRewards.sol — emitUserLoyaltyRewardBatch()

Impact

Admin can call with arbitrary parameters (any user, any amountDeposited, any ksuTokenPrice) without on-chain validation against actual state. A compromised admin multisig could inflate rewards and drain all KSU from the contract.

Root Cause

Batch emission function trusts admin input without on-chain state validation.

Recommended Fix

Add a maximum reward cap per call. Ensure admin is a well-configured multisig with timelock. Document as intentional admin privilege.

M-04 MEDIUM Stale KSU epoch price inflates loyalty levels when updateKsuEpochTokenPrice not called
Contract

SystemVariables.sol / UserManager.sol

Impact

updateKsuEpochTokenPrice() must be called each epoch by an external caller. If not called, ksuEpochTokenPrice remains at its last-updated value. A stale high price inflates loyalty levels, giving unfair clearing priority and higher KSU rewards.

Root Cause

No enforcement that updateKsuEpochTokenPrice() is called every epoch. No automated mechanism.

Recommended Fix

Call updateKsuEpochTokenPrice() at the start of batchCalculateUserLoyaltyLevels, or use the live price directly.

M-06 MEDIUM Incomplete loss token minting blocks clearing step 4
Contract

LendingPoolTrancheLoss.sol / LendingPoolTranche.sol

Impact

When a loss is reported with doMintLossTokens = false, the notPendingLossMint modifier blocks all deposit() and redeem() on the tranche until batch minting completes. Clearing step 4 calls these functions, causing clearing to revert with LossMintingInProgress.

Root Cause

ClearingCoordinator.doClearing does not check for pending loss mints before attempting step 4.

Recommended Fix

Require doMintLossTokens = true for production usage. Document operational guidelines for the parameter.

M-08 MEDIUM Withdrawal pro-rata truncation to zero permanently delays small positions
Contract

AcceptedRequestsExecution.sol

Impact

When computing accepted withdrawal shares pro-rata, small positions truncate to 0 due to integer division. The withdrawal NFT survives with full shares amount, and the user gets 0 shares accepted. This repeats every epoch — the request is effectively stuck permanently.

Root Cause

Integer division truncation with no minimum acceptance threshold or dust-clearing mechanism.

Recommended Fix

Set a minimum accepted amount, round up for smallest positions, or allow users to cancel and re-submit.

Go / No-Go Recommendation

Recommendation: CONDITIONAL GO

The Kasu protocol has a well-engineered architecture with strong defense-in-depth patterns. The permissioned, epoch-based design eliminates most common DeFi attack vectors. No critical or high-severity vulnerabilities remain. Five findings (1 High, 4 Medium) were resolved during this audit cycle with verified fixes. However, the following conditions should be met:

Should Fix (before next audit cycle)

IDFindingRationale
M-03Batch reward inflationAdmin privilege risk. Consider max reward cap or timelock.
M-04Stale KSU epoch priceOperational risk. Consider automating price update in clearing flow.
M-06Loss minting blocks clearingOperational DoS risk. Document guidelines for doMintLossTokens.
M-08Withdrawal truncationAffects small positions. Consider dust-clearing mechanism.

Positive Observations

  • ERC4626 inflation well-mitigated — 12-decimal offset makes inflation attacks economically infeasible.
  • Comprehensive verification in clearing — _verifyResult in AcceptedRequestsCalculation provides thorough post-computation invariant checking.
  • Well-designed batch processing — Steps 1, 2, and 4 support batching to prevent gas-limit DoS.
  • Proper use of OpenZeppelin v5 — Per-proxy ProxyAdmin, SafeERC20, AccessControl, ERC4626 with virtual assets.
  • Epoch-based design eliminates MEV — No real-time pricing, no flash loan vectors, no sandwich attacks.
  • Clean Lite variant architecture — Lite contracts properly stub out token-dependent functionality while preserving KYC and core deposit flows.
  • Strong coupled state consistency — Of 15 mapped coupled state pairs, the only desynchronization (H-01) has been fixed.
  • Non-transferable tokens — PendingPool NFTs, tranche shares, and loss tokens are all correctly restricted.
  • Responsive remediation — 5 findings resolved during audit cycle with verified fixes and dedicated test coverage.
